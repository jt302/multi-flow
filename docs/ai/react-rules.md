# React 开发规范

本规范用于指导 AI agent 在本项目中编写、修改和重构 React 代码。
目标是让生成结果保持：**一致、可维护、可测试、易审查**。

---

## 1. 总体原则

### 1.1 优先级

Agent 在做 React 开发时，优先遵循以下顺序：

1. **不破坏现有行为**
2. **保持与现有代码风格一致**
3. **优先可读性，其次才是“聪明”的写法**
4. **优先小步修改，不做无关重构**
5. **优先组合而非复杂抽象**

### 1.2 基本要求

- 只修改完成任务所必须的文件和逻辑。
- 不要顺手重命名大量变量、调整无关格式、移动无关文件。
- 不要引入新依赖，除非任务明确要求且理由充分。
- 优先复用项目中已有组件、hooks、工具函数、样式模式。
- 遇到不确定的业务逻辑时，保守处理，避免擅自改变用户可见行为。

---

## 2. 组件设计规则

### 2.1 组件职责

每个组件应尽量只做一件事：

- **UI 展示组件**：负责渲染
- **容器组件 / 页面组件**：负责数据组装、状态协调
- **通用组件**：负责可复用交互逻辑

避免一个组件同时承担：

- 数据请求
- 大量业务计算
- 复杂状态机
- 大片 JSX 展示
- DOM 细节控制

如果一个组件超过以下任一条件，应考虑拆分：

- 文件明显超过 200~300 行
- `return` 中 JSX 层级很深
- 同时管理 5 个以上状态
- 存在多个互相独立的交互区域
- 阅读时需要频繁上下跳转理解逻辑

### 2.2 组件拆分原则

优先按以下维度拆分：

- 按视觉区块
- 按业务职责
- 按交互单元
- 按复用潜力

不要为了“拆分而拆分”。
拆出来的子组件必须让主组件更清晰，而不是让代码跳转更多、理解更难。

### 2.3 组件定义方式

- 使用 **函数组件**
- 除非项目已有明确约定，否则不要使用 class component
- 组件导出方式保持与项目现有风格一致
- 默认优先具名组件，便于调试和搜索

---

## 3. Props 规范

### 3.1 Props 设计

- Props 名称必须清晰、语义化
- 避免把过多零散参数塞进组件
- 参数过多时，优先审视组件职责是否过重
- 布尔 props 命名应体现状态或能力，例如：
  - `disabled`
  - `loading`
  - `open`
  - `readonly`

避免模糊命名：

- `data`
- `info`
- `item`（除非上下文非常明确）
- `handleSomething`（可更具体）

### 3.2 Props 使用

- 不要无脑透传所有 props 到下层组件
- 不要把未使用的 props 留在接口里
- 不要为了省事把整个大对象传下去，除非确实合理
- 子组件只接收它真正需要的数据和回调

---

## 4. State 管理规范

### 4.1 本地状态优先

状态优先级：

1. 能用局部变量解决，就不用 state
2. 能由 props / selector 推导，就不要重复存 state
3. 只在组件内部使用的状态，放本地
4. 跨组件共享且有明确必要时，再提升或放全局

### 4.2 避免冗余状态

不要保存可以直接推导出的值，例如：

- 已经能从 props 计算出的派生值
- 已有数组过滤/映射后的结果
- 能从 URL、路由、查询参数中直接获得的信息

### 4.3 状态更新

- 状态命名要清楚表达含义
- 多个强相关状态，优先考虑合并管理
- 复杂交互优先考虑 `useReducer`
- 避免链式、隐式、难跟踪的状态更新

---

## 5. Hooks 规范

### 5.1 基本规则

- 严格遵守 Hooks 规则
- 不要在条件、循环、嵌套函数中调用 hooks
- 不要为了“代码好看”破坏 hook 顺序

### 5.2 useEffect 使用原则

`useEffect` 只用于：

- 副作用
- 外部系统同步
- 订阅 / 解绑
- 请求触发
- DOM / 浏览器 API 协调

不要用 `useEffect` 做：

- 纯数据计算
- 可以在 render 阶段直接得到的派生值
- 为了“监听某值然后 setState”而制造冗余状态

优先判断：

- 这是不是副作用？
- 这是不是其实应该写成普通变量、`useMemo` 或直接内联表达式？

### 5.3 自定义 hooks

当以下逻辑重复出现时，提取自定义 hook：

- 复用状态逻辑
- 复用副作用逻辑
- 复用数据获取与状态协调
- 复用事件监听 / 生命周期封装

自定义 hook 命名必须以 `use` 开头，并保证：

- 输入明确
- 输出稳定
- 不泄漏不必要实现细节

---

## 6. 事件处理规范

### 6.1 命名

事件处理函数优先使用以下命名：

- `handleClick`
- `handleSubmit`
- `handleChange`
- `handleClose`

传给子组件的回调优先使用：

- `onClick`
- `onSubmit`
- `onChange`
- `onClose`

约定：

- `onXxx` 表示组件对外暴露的回调
- `handleXxx` 表示组件内部实现函数

### 6.2 事件逻辑

- 事件处理函数应尽量短小
- 复杂逻辑提取到独立函数或 hook
- 不要在 JSX 内联大段匿名函数逻辑
- 不要把大量业务流程直接塞进按钮点击回调中

---

## 7. 数据获取与异步处理

### 7.1 数据请求

- 数据请求逻辑不要散落在多个 UI 组件中
- 优先集中在页面层、容器层或专用 hooks 中
- 展示组件尽量不要直接发请求，除非它本身就是独立数据组件

### 7.2 异步状态

所有异步流程尽量明确区分：

- loading
- success
- empty
- error

不要只处理成功态。
必须考虑：

- 首次加载
- 加载失败
- 无数据
- 重试
- 中断 / 卸载时的安全性

### 7.3 容错

- 不要吞掉错误
- 错误信息至少应可追踪
- 用户可见错误文案要可理解
- 开发日志和用户提示要区分

---

## 8. 目录与文件组织

### 8.1 文件职责

一个文件只承载一个主要职责，常见分类：

- `components/`：通用组件
- `pages/`：页面级组件
- `hooks/`：自定义 hooks
- `utils/`：纯工具函数
- `services/`：接口请求、服务封装
- `types/`：类型定义
- `constants/`：常量
- `stores/`：全局状态管理

### 8.2 文件放置原则

- 组件、样式、测试、类型，尽量就近组织
- 高频协作模块保持扁平易找
- 不要制造过深目录嵌套
- 不要把所有类型都堆到一个巨大 `types.ts` 中

---

## 9. 命名规范

### 9.1 通用命名

- 组件：`PascalCase`
- hooks：`useXxx`
- 工具函数：`camelCase`
- 常量：`UPPER_SNAKE_CASE`
- 布尔变量：用 `is`、`has`、`can`、`should` 开头
- 事件回调：`onXxx`
- 内部处理函数：`handleXxx`

### 9.2 命名要求

命名应直接表达业务语义。避免：

- `temp`
- `data`
- `obj`
- `list2`
- `newItem`
- `finalData`

变量名长度应与作用域匹配：

- 作用域很小，可适度简短
- 跨函数、跨组件传递时必须清晰

---

## 10. JSX 与渲染规范

### 10.1 JSX 书写原则

- 保持 JSX 简洁
- 复杂判断提前到 JSX 外整理
- 避免三层以上嵌套三元表达式
- 多分支渲染优先使用清晰的条件块或抽取函数

### 10.2 条件渲染

允许：

- 简单 `&&`
- 简单三元表达式

避免：

- 多重嵌套三元
- 在 JSX 里塞很长的逻辑判断
- 通过隐式真假值制造难懂行为

### 10.3 列表渲染

- 必须提供稳定 `key`
- 禁止在可重排列表中随意使用 index 作为 key
- key 应来自业务唯一标识

---

## 11. 样式规范

### 11.1 样式原则

- 保持与项目现有样式方案一致
- 优先复用现有 design tokens / 组件变体 / 工具类
- 不要为了单次需求引入新的样式体系

### 11.2 Tailwind / 原子类使用

如果项目使用 Tailwind：

- 优先复用已有 class 组合模式
- 重复样式可提取为组件、变体或工具函数
- 不要在一个元素上堆过长且混乱的 class 串
- 状态类、布局类、视觉类应尽量有基本组织

### 11.3 样式禁忌

- 不要滥用内联样式
- 不要硬编码魔法数字，除非确有必要
- 不要随意覆盖全局样式
- 不要写与当前任务无关的大规模视觉重构

---

## 12. 可访问性规范

Agent 生成 UI 时必须默认考虑基本可访问性：

- 使用正确的语义化标签
- 按钮用 `button`
- 可点击但非导航的元素不要滥用 `div`
- 表单控件需要可关联的 label
- 图标按钮应提供可访问名称
- 键盘操作不能被破坏
- 弹层、对话框、菜单要考虑焦点管理
- 不要只依赖颜色表达状态

---

## 13. TypeScript 规范（如果项目启用 TS）

### 13.1 类型使用原则

- 给公开接口、props、函数参数、返回值提供清晰类型
- 优先使用明确类型，而不是 `any`
- 仅在无法避免时使用 `unknown` + 类型收窄
- 避免滥用类型断言

### 13.2 类型设计

- 类型名应体现业务语义
- 公共类型复用，局部类型就近定义
- 不要把超复杂内联类型直接塞进组件 props 中
- 复杂对象类型可单独提取

### 13.3 禁止事项

- 禁止无理由新增 `any`
- 禁止用断言掩盖真实类型问题
- 禁止为了通过编译而牺牲类型可信度

---

## 14. 测试与可测试性

### 14.1 可测试性原则

写代码时应让其易于测试：

- 业务逻辑与 UI 尽量分离
- 纯函数保持纯净
- 副作用边界明确
- 复杂条件尽量拆出独立逻辑

### 14.2 Agent 修改代码时

如果任务涉及：

- 表单流程
- 核心交互
- 条件分支修复
- 数据处理逻辑

则优先考虑是否需要：

- 补充测试
- 更新已有测试
- 至少保证现有测试不失效

---

## 15. 性能规范

### 15.1 性能原则

- 不要过早优化
- 但也不要无视明显低效写法
- 优先修正“明显不合理”的重复渲染和重复计算

### 15.2 useMemo / useCallback

- 仅在有明确收益时使用
- 不要把它们当默认写法
- 不要为了“看起来专业”把所有函数都包一层

### 15.3 大列表与重渲染

遇到大列表、复杂面板、频繁交互区域时，应关注：

- 不必要重渲染
- 过重计算放在 render 中
- key 不稳定
- 父组件状态变化导致整棵子树频繁刷新

---

## 16. 日志、提示与错误处理

### 16.1 日志

- 调试日志不要遗留在最终代码中
- 生产代码中的日志必须有明确价值
- 不要输出敏感信息

### 16.2 用户提示

- 成功、失败、警告文案应清晰
- 不要把底层报错原文直接暴露给用户
- 提示文案优先可理解，其次才是技术准确性

---

## 17. 修改现有代码时的行为规范

Agent 修改现有 React 代码时必须遵守：

- 先理解原有模式，再动手
- 优先最小改动达成目标
- 不要顺手重构整个文件
- 不要把“修 bug”变成“重写模块”
- 不要替换成熟模式为个人偏好写法
- 不要因为局部不喜欢就大规模改命名、改结构、改格式

如果发现明显问题但不属于当前任务：

- 可在注释或总结中指出
- 不要未经要求直接扩大改动范围

---

## 18. 禁止事项

以下行为默认禁止：

- 无理由引入新状态管理方案
- 无理由引入新 UI 库
- 无理由新增全局副作用
- 在组件里写超长业务流程
- 用 `any` 掩盖问题
- 用 index 作为动态列表 key
- 在 `useEffect` 中做本可直接推导的逻辑
- 把一个页面的所有内容堆进单文件
- 为了抽象而抽象
- 为了“现代化”而重写稳定代码

---

## 19. 提交结果时的输出要求

Agent 在完成 React 相关任务时，应尽量说明：

1. 修改了哪些文件
2. 为什么这样改
3. 是否影响现有行为
4. 是否存在风险点
5. 后续还可优化什么（仅列与当前任务强相关内容）

不要给出空泛总结，必须结合实际改动说明。

---

## 20. 推荐决策准则

当有多个实现方案时，优先选择：

- 更贴近现有项目风格的方案
- 更容易 review 的方案
- 更容易回滚的方案
- 更少副作用的方案
- 更少隐式行为的方案
- 更容易被下一个开发者接手的方案

---

## 21. 一句话准则

> 写出来的 React 代码必须让人一眼看懂“这个组件负责什么、数据从哪里来、交互如何发生、为什么这样写”。
